Parcours Exercices Elasticsearch Shakespeare

Accueil Hub exos

Objectif: pratiquer un workflow complet Elasticsearch (intro, approfondissement, full-text, aggregations, niveau avance) avec des consignes explicites et un resultat attendu pour chaque exercice.

Bloc 0 - Introduction (issus du Cours 02)

A) Creer l'index shakespeare avec le mapping du cours

Objectif: valider la base du modele (types exacts pour recherche textuelle + filtres).

Resultat attendu: index cree, champs numeriques en integer, champs exacts en keyword, texte en text.

Voir la correction
PUT shakespeare
{
  "mappings": {
    "properties": {
      "line_id": { "type": "integer" },
      "play_name": { "type": "keyword" },
      "speech_number": { "type": "integer" },
      "line_number": { "type": "keyword" },
      "speaker": { "type": "keyword" },
      "text_entry": { "type": "text" }
    }
  }
}

B) Charger les donnees avec la Bulk API

Objectif: importer rapidement le dataset Shakespeare en format NDJSON.

Resultat attendu: import sans erreur, puis count > 0 apres refresh.

Voir la correction
curl -L "https://raw.githubusercontent.com/grokify/kibana-tutorial-go/refs/heads/master/shakespeare.json" -o sandbox/data/shakespeare.json

curl -X POST "http://localhost:9200/shakespeare/_bulk?pretty" \
  -H "Content-Type: application/x-ndjson" \
  --data-binary @sandbox/data/shakespeare.json

curl -X POST "http://localhost:9200/shakespeare/_refresh"
curl "http://localhost:9200/shakespeare/_count?pretty"

E) Retrouver les citations de Romeo and Juliet qui parlent de sun

Objectif: combiner recherche textuelle (match) et filtre metier exact (term).

Resultat attendu: documents de la piece cible contenant l'idee de \"sun\".

Voir la correction
GET shakespeare/_search
{
  "query": {
    "bool": {
      "must": [
        { "match": { "text_entry": "sun" } }
      ],
      "filter": [
        { "term": { "play_name": "Romeo and Juliet" } }
      ]
    }
  }
}

Bloc 0B - Aller plus loin (issus du Cours 02B)

F) Identifier si une requete doit utiliser match ou term

Objectif: choisir le bon type de requete selon le type de champ et le besoin metier.

Resultat attendu: match pour text_entry, term pour speaker / play_name.

Voir la correction
# plein texte
GET shakespeare/_search
{
  "query": { "match": { "text_entry": "question" } }
}

# exact
GET shakespeare/_search
{
  "query": { "term": { "speaker": "HAMLET" } }
}

G) Ecrire une requete bool avec must + filter

Objectif: separer la pertinence textuelle de la contrainte metier exacte.

Resultat attendu: match sur text_entry + filtre exact sur play_name.

Voir la correction
GET shakespeare/_search
{
  "query": {
    "bool": {
      "must": [
        { "match": { "text_entry": "love" } }
      ],
      "filter": [
        { "term": { "play_name": "Othello" } }
      ]
    }
  }
}

H) Ajouter une clause should pour booster certains resultats

Objectif: comprendre qu une requete peut preferer certains documents sans les rendre obligatoires.

Resultat attendu: documents Hamlet favorisant ceux contenant question ou time.

Voir la correction
GET shakespeare/_search
{
  "query": {
    "bool": {
      "must": [
        { "term": { "play_name": "Hamlet" } }
      ],
      "should": [
        { "match": { "text_entry": "question" } },
        { "match": { "text_entry": "time" } }
      ]
    }
  }
}

I) Tester une modelisation multi-fields pour speaker

Objectif: permettre recherche texte sur speaker et filtre exact sur speaker.keyword.

Resultat attendu: mapping avec speaker en text + sous-champ keyword.

Voir la correction
PUT shakespeare_multifield
{
  "mappings": {
    "properties": {
      "speaker": {
        "type": "text",
        "fields": {
          "keyword": { "type": "keyword" }
        }
      },
      "text_entry": { "type": "text" }
    }
  }
}

GET shakespeare_multifield/_search
{
  "query": {
    "term": { "speaker.keyword": "HAMLET" }
  }
}

J) Construire un index avec normalizer lowercase sur un champ keyword

Objectif: rendre les filtres exacts plus coherents sur les variations de casse.

Resultat attendu: index cree avec normalizer, puis requetes exactes plus robustes.

Voir la correction
PUT shakespeare_norm
{
  "settings": {
    "analysis": {
      "normalizer": {
        "lowercase_normalizer": {
          "type": "custom",
          "filter": ["lowercase"]
        }
      }
    }
  },
  "mappings": {
    "properties": {
      "speaker": {
        "type": "keyword",
        "normalizer": "lowercase_normalizer"
      },
      "text_entry": { "type": "text" }
    }
  }
}

K) Choisir une strategie quand le mapping initial est mauvais

17) Requete booleenne avec exclusion de piece

Objectif: combiner must et must_not proprement.

Resultat attendu: lignes sur le theme "night" sans la piece exclue.

Voir la solution

{

  }

18) Mettre en place une pagination scalable avec search_after

Objectif: paginer sans utiliser from sur un grand volume.

Resultat attendu: page 2 obtenue a partir du tuple de tri du dernier hit de la page 1.

Voir la solution
# page 1
GET shakespeare/_search
{
  "size": 3,
  "sort": [{"speech_number":"asc"}, {"line_id":"asc"}],
  "query": {"match_all": {}}
}

# page 2 (exemple)
GET shakespeare/_search
{
  "size": 3,
  "sort": [{"speech_number":"asc"}, {"line_id":"asc"}],
  "search_after": [1, 45],
  "query": {"match_all": {}}
}

19) Creer un runtime field line_length et calculer sa moyenne par piece

Objectif: enrichir a la volée sans reindexation.

Resultat attendu: metrique avg_line_length disponible dans chaque bucket de piece.

Voir la solution
GET shakespeare/_search
{
  "size": 0,
  "runtime_mappings": {
    "line_length": {
      "type": "long",
      "script": "emit(params._source.text_entry != null ? params._source.text_entry.length() : 0)"
    }
  },
  "aggs": {
    "by_play": {
      "terms": { "field": "play_name", "size": 10 },
      "aggs": {
        "avg_line_length": { "avg": { "field": "line_length" } }
      }
    }
  }
}
