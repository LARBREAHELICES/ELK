Exercices Logstash

Accueil Hub exos

Objectif: construire un pipeline Logstash robuste de bout en bout (input, parsing, enrichissement, sortie Elasticsearch, verification runtime).

1) Generer un fichier de logs de test

Objectif: disposer d'un jeu d'entree deterministe avec plusieurs niveaux de logs.

Resultat attendu: un fichier logs/app.log avec des lignes ISO8601 parseables.

Voir la solution
cat > logs/app.log <<'LOG'
2026-03-30T10:15:00Z INFO api - User created order
2026-03-30T10:16:45Z WARN billing - Payment retry
2026-03-30T10:17:12Z ERROR worker - Timeout
LOG

2) Configurer un input file fiable pour le TP

Objectif: relire le fichier depuis le debut a chaque run (pas d etat sincedb persistant).

Resultat attendu: les 3 lignes sont lues meme apres restart du pipeline.

Voir la solution
input {
  file {
    path => "/usr/share/logstash/logs/*.log"
    start_position => "beginning"
    sincedb_path => "/dev/null"
  }
}

3) Parser chaque ligne avec grok

Objectif: extraire log_timestamp, level, service, log_message.

Resultat attendu: plus de champ message brut uniquement; les champs metier existent.

Voir la solution
filter {
  grok {
    match => {
      "message" => "%{TIMESTAMP_ISO8601:log_timestamp} %{LOGLEVEL:level} %{DATA:service} - %{GREEDYDATA:log_message}"
    }
    tag_on_failure => ["_grokparsefailure_app"]
  }
}

4) Normaliser la date applicative vers @timestamp

Objectif: rendre les logs exploitables directement dans Discover et les visualisations temporelles.

Resultat attendu: @timestamp correspond a log_timestamp.

Voir la solution
date {
  match => ["log_timestamp", "ISO8601"]
}

5) Ajouter un enrichissement simple de severite

Objectif: mapper le niveau de log vers une priorite numerique utile en tri/aggregation.

Resultat attendu: nouveau champ severity_rank present sur chaque evenement.

Voir la solution
mutate {
  add_field => { "severity_rank" => 1 }
}
if [level] == "WARN" {
  mutate { replace => { "severity_rank" => 2 } }
}
if [level] == "ERROR" {
  mutate { replace => { "severity_rank" => 3 } }
}
mutate {
  convert => { "severity_rank" => "integer" }
}

6) Sortir les evenements vers Elasticsearch

Objectif: indexer quotidiennement dans app-logs-YYYY.MM.dd.

Resultat attendu: creation automatique d'un index app-logs-*.

Voir la solution
output {
  elasticsearch {
    hosts => ["http://elasticsearch:9200"]
    index => "app-logs-%{+YYYY.MM.dd}"
  }
}

7) Verifier l'ingestion cote Elasticsearch

Objectif: confirmer volume, structure et contenu avant de passer a Kibana.

Resultat attendu: count coherent avec le nombre de lignes injectees.

Voir la solution
curl "http://localhost:9200/_cat/indices?v"
curl "http://localhost:9200/app-logs-*/_count?pretty"
curl "http://localhost:9200/app-logs-*/_search?pretty"

8) Isoler les erreurs par service avec aggregation

Objectif: produire une vue exploitable pour prioriser les incidents.

Resultat attendu: buckets par service pour level=ERROR.

Voir la solution
GET app-logs-*/_search
{
  "query": { "term": { "level": "ERROR" } },
  "aggs": {
    "by_service": { "terms": { "field": "service.keyword" } }
  }
}

9) Detecter les lignes mal parsees

Objectif: surveiller la qualite du parsing grok en production.

Resultat attendu: 0 hit sur le tag de parse failure si tout est correct.

Voir la solution
GET app-logs-*/_search
{
  "query": {
    "term": { "tags.keyword": "_grokparsefailure_app" }
  }
}

10) Verifier la sante runtime de Logstash

Objectif: valider que le pipeline tourne sans crash ni erreur bloquante.

Resultat attendu: endpoint 9600 repond et logs conteneur sans erreur critique.

Voir la solution
curl "http://localhost:9600"
docker compose logs -f logstash
